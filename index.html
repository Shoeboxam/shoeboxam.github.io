EndSearchSolutionsRequest

Ends the search and releases all resources associated with the solution search.

If the call is made in parallel with a running search and results are being streamed,

the search is stopped and the "GetSearchSolutionsResults" stream is closed by an AutoML system

(as happens when the search is concluded on its own, or when a search is stopped

by "StopSearchSolutions"). Found solution IDs during the search are no longer valid

after this call.

FitSolutionRequest

Fit the solution on given inputs. If a solution is already fitted on inputs this is a NOOP

(if no additional outputs should be exposed). This can happen when a TA2 simultaneously

fits the solution as part of the solution search phase.

FieldTypeLabelDescription

inputs Value repeated

expose_outputs string repeated

List of data references of step outputs which should be exposed to the TA3 system. If you want to expose outputs of the whole pipeline (e.g., predictions themselves), list them here as well. These can be recursive data references like "steps.1.steps.4.produce" to point to an output inside a sub-pipeline. Data references for step outputs which otherwise would not be computed during pipeline execution can also be provided to force them to be computed and exposed.

FieldTypeLabelDescription
search_id string

GetSearchSolutionsResultsResponse

FieldTypeLabelDescription
progress Progress

Overall process progress, not progress per solution. While solutions are being found and returned, or scores computed and updated, progress state should be kept at "RUNNING".

done_ticks double

A measure of progress during search. It can be any number of internal steps or actions an AutoML system is doing during search. It can be even number of how many candidate solutions were already examined. It does not even have to be an integer. How regularly a change to this number is reported to a client is left to an AutoML system's discretion, but a rule of thumb is at least once a minute if the number changes.

all_ticks double

If an AutoML system knows how many internal steps or actions are there, it can set this field. This can also be updated through time if more (or even less) internal steps or actions are determined to be necessary. If this value is non-zero, then it should always hold that "done_ticks" <= "all_ticks".

solution_id string

internal_score double

Internal score for this solution between 0.0 and 1.0 where 1.0 is the highest score. There is no other meaning to this score and it does not necessary depend on scores listed in the problem description. Optional. Because this field is optional, if omitted the default value will be 0. But 0 is a valid value for this field. Because of that you should never omit the field. If you do not have internal score to provide, use NaN for the value of this field to signal that. When solution cannot be scored (i.e., the fully specified pipeline used as a template is not a standard pipeline) internal score does not have to be provided.

scores SolutionSearchScore repeated

An AutoML system might be able to provide more meaningful scores as well, depending on its approach to solution search. Moreover, even the same AutoML system might not use the same scoring approach for all of its solutions. Optional. When solution cannot be scored (i.e., the fully specified pipeline used as a template is not a standard pipeline) score does not have to be provided.

List all primitives known to TA2, their IDs, versions, names, and digests. Using this

information a TA3 should know which primitives may be put into a pipeline template.

ScoringConfiguration

FieldTypeLabelDescription
method string

The evaluation method to use. Standard values are required to be supported and are: * "HOLDOUT" * "K_FOLD" In addition, "RANKING" evaluation method should be also supported. "RANKING" evaluation method can be used for a rank of a solution among all found solutions of a given "SearchSolutions" call. Invalid to use before the "SearchSolutions" call, of which the solution is part of, has successfully completed or has been stopped. Only possible with "RANK" performance metric. The following additional evaluation methods are defined to allow expressing internal evaluation methods used by an AutoML system during solution search. If any method being used is missing, feel free to request it to be added. * "LEAVE_ONE_OUT" * "PREDICTION" Instead of really scoring, an AutoML system might predict the score only. * "TRAINING_DATA" Training data is reused to test as well. No standard value will ever start with "_", so if you worry about potential conflicts with future standard values, consider starting your non-standard values with "_'.

folds int32

Number of folds made, if applicable.

train_test_ratio double

If applicable, the ratio between the train and test data and represents the proportion of the input data to include in the train split. The rest is included in the test split. E.g., "train_test_ratio == 0.8" would mean that 80% of the whole data becomes the train split and 20% of the whole data becomes the test split.

shuffle bool

Shuffle data? Set to true if employed.

random_seed int32

Value for random seed to use for shuffling. Note, not setting this value means always using seed of 0.

stratified bool

Do stratified k-fold? Set to true if employed.

SearchSolutionsRequest

Starts a new solution search. Found solutions have not necessary been fitted on the provided

inputs. Problem description and inputs are used only to help guide the search process.

Consider any found solutions to be just a static description of solutions at this stage.

Multiple parallel solution searches can happen at the same time.

FieldTypeLabelDescription
user_agent string

Some string identifying the name and version of the AutoML system.

version string

Shall be set to "protocol_version" above.

time_bound_search double

Desired upper limit of time for solution search, expressed in minutes. Default value of 0 (and any negative number) signifies no time bound. See also time_bound_run.

priority double

Value stating the priority of the search. If multiple searches are queued then highest priority (largest number) should be started next by the AutoML system. Primarily used to sort any queue, but no further guarantee that the AutoML system can give more resources to high priority searches. If unspecified, by default search will have priority 0. Negative numbers have still lower priority.

allowed_value_types string repeated

Which value types can an AutoML system use to communicate values to a client? The order is important as the AutoML system will try value types in order until one works out, or an error will be returned instead of the value.

problem ProblemDescription

Problem description to use for the solution search. Problem description is optional with fully specified pipelines which the client provides. If still provided, an AutoML system should use it to set semantic types on target columns and any other use of a problem description for the needs of running a pipeline.

template PipelineDescription

A pipeline template to use for search or to execute. If template is omitted, then a regular solution search is done. If template consists only of one placeholder step, then a regular solution search is done to replace that step. If there is no placeholder step, but template describes a full pipeline with free hyper-parameters, then this call becomes a hyper-parameter tuning call over free hyper-parameters and found solutions share the same pipeline, but different hyper-parameter configurations (unless "use_default_values_for_free_hyperparams" is set to true). If there is no placeholder step and all hyper-parameters are fixed as part of the pipeline, then this call only checks the given template and returns the solution with same pipeline back, to be executed. This allows fixed computations to be done on data, for example, pipeline can consist of only one primitive with fixed hyper-parameters to execute that one primitive. Moreover, such fully specified pipelines with fixed hyper-parametres can have any inputs and any outputs. Otherwise pipelines have to be from a Dataset container value to predictions Pandas dataframe. While there are all these options possible, only a subset has to be supported by all systems: - Omitted templates, - Partial pipelines with one placeholder step, at the last step in the pipeline template, - Fully specified pipelines without free hyper-parameters, or with free hyper-parameters and "use_default_values_for_free_hyperparams" set to true. See the Pipeline section of the README for more details.

inputs Value repeated

Pipeline inputs used during solution search. They have to point to Dataset container values. Order matters as each input is mapped to a template's input in order. Optional for templates without a placeholder and with all hyper-parameters fixed.

time_bound_run double

Desire upper limit of time for one pass of the pipeline run expressed in minutes. Expressed at time of search, an AutoML system should limit the search to solutions that would not take longer than this for one pass of the pipeline run. Default value of 0 (and any negative number) signifies no time bound. This can also functionally be used as the time bound for scoring also.

rank_solutions_limit int32

Suggested maximum number of solutions to rank. Default is 0. If it is 0, it means no ranking of found solutions is done and no limit on number of found solutions is set. An AutoML system can still provide "RANK" score as part of "scores" field if ranking approach the AutoML system uses still allows to stream found solutions as they are found. If value of this field is not 0, the behavior of this call changes: - "time_bound_search" becomes a time bound for both search and ranking. The AutoML system decides how to divide the time between searching and ranking. - The AutoML system is required to provide "RANK" score as part of "scores" field in messages "GetSearchSolutionsResultsResponse". - The AutoML system can make all found solutions available only at the end of the call. If ranking approach used by the AutoML system allows for that, the AutoML system can still stream found solutions (and their ranks) during whole call. - Calling "StopSearchSolutions" stops both search and ranking processes, but the AutoML system can quickly return ranked solutions up to that point, if possible, before closing the "GetSearchSolutionsResults" stream. Similarly for "EndSearchSolutions", but that is less useful. - The AutoML system can still return more solutions than the limit. The limit only signals that the AutoML system does not have to rank and return more solutions than the limit, which is useful when ranking is an expensive operation.

random_seed int32

Random seed used to initiate search. Note, not setting this value means always using seed of 0.

use_default_values_for_free_hyperparams bool

If set to true, during search an AutoML system is suggested to not do hyper-parameter tuning of free hyper-parameters but use their default values. It might be that the AutoML system operates searches for both the pipeline and hyper-parameter values at the same time and flag cannot be easily respected, in that case this flag can be ignored by the AutoML system. When template is provided and contains free hyper-parameters but no placeholders, then setting this flag to true means that for all free hyper-parameters in the template default values should be used. This allows a client to not have to fix values for all free hyper-parameters in the template itself, to obtain a fully specified pipelines without free hyper-parameters. Generally applies only to tuning hyper-parameters.

search_features string repeated

Suggested search features to use to an AutoML system, from ones supported by the AutoML system as provided in "supported_search_features" of the "HelloResponse". The AutoML system can ignore any suggestion. When not provided or when ignored, the AutoML system can use any search features, or even search features not listed in "supported_search_feature".

PipelineDescription

Pipeline description matches the D3M pipeline description.

It serves two purposes: describing found pipelines by an AutoML system to a client, and communicating

pipeline templates by a client to an AutoML system. Because of this some fields are reasonable only in

one of those uses. They are marked with "AutoML" or "client" in the comment, for fields which are primarily

to be set only by an AutoML system or only by a client, respectively.

FieldTypeLabelDescription
id string

AutoML: UUID of the pipeline. Templates do not have IDs. But a client might provide it for a fully specified pipeline. It does not necessary have to match "solution_id" from "ListSolutionsResponse" and other related messages. Those IDs are about whole solutions (pipeline, potentially fitted, with set hyper-parameters). This here ID is about this particular ID description.

source PipelineSource

"schema" field is not needed because it is fixed by the D3M AutoML RPC protocol version. System which generated a pipeline or a template. Optional.

created google.protobuf.Timestamp

AutoML: Timestamp when created. Templates do not have this timestamp. A cleint might provide it for a fully specified pipeline.

name string

Human friendly name of the pipeline. For templates it can be a hint to an AutoML system how to name found pipelines. Optional.

description string
